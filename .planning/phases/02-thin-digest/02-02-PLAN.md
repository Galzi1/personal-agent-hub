<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| pipeline.py → Discord REST API | Outbound HTTP; bot token in Authorization header; message content derived from untrusted RSS titles |
| MicroClaw stdout → MicroClaw post behavior | MicroClaw may relay stdout to Discord; prompt update prevents double-post |
| config/microclaw.config.yaml → pipeline.py | Secret file on disk; channel_id and bot_token read at runtime |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | Tampering | pipeline.py:run_digest Step 6 | mitigate | RSS-sourced content enters Discord only via format_digest(), which truncates titles at 80 chars. Angle brackets wrap URLs suppressing embeds. No user-controlled input reaches the Discord payload directly. |
| T-02-05 | Information Disclosure | pipeline.py:run_digest / logger.error | mitigate | logger.error logs the exception string from httpx.HTTPStatusError (HTTP status line only). discord_token is a local variable passed as argument - it does not appear in any exception message or log call. Verified: the only log statement is `f"Discord API error after {posted}/{len(messages)} messages: {e}"`. |
| T-02-06 | Denial of Service | pipeline.py:run_digest Discord posting loop | accept | Phase 2 produces 1-2 messages per run. Discord rate limit (5 msg/5s per channel) is not approached. Explicit rate-limit handling deferred per CONTEXT.md. |
| T-02-07 | Tampering | MicroClaw scheduled task prompt | mitigate | Prompt updated (Task 2) to prevent MicroClaw from posting pipeline stdout to Discord. Without this update, the confirmation string becomes a second unsolicited Discord message. Human-verified in checkpoint. |
</threat_model>
